# Phase 1.6 — Pre-Phase-2 Hardening: Design Spec

> **Status:** approved design, pre-implementation.
> **Date:** 2026-05-29.
> **Author:** Leo Chang.
> **Gate:** This phase is a hard prerequisite for Phase 2 (LLM agent layer), inserted by advisor (Yongfeng) on 2026-05-26.

## Motivation

The Phase 1 simulator is complete and validated, but the headline scenario (12 h
overnight July outage on Texas ResStock data) has a no-coordination baseline that
already serves 98.5% of load. That ceiling leaves only ~0.5% of headroom for any
coordination layer to demonstrate value — round-robin reaches 99.5%, and a Phase 2
LLM layer would be fighting over the remaining half-percent. Building the LLM layer
against that regime risks a null result by construction.

The advisor required four things before Phase 2 begins:

1. A **realistically-motivated communication graph** — coordination edges that follow
   ownership and management relationships, not just spatial adjacency, producing
   multiple partially-overlapping "trust circles."
2. **Stress scenarios** where simple round-robin sharing visibly breaks (substantial
   unmet demand and unequal welfare remain).
3. A **centralized LP upper-bound baseline** so results can be framed as "coordination
   closes X% of the gap between round-robin and LP-optimal."
4. **One or two additional public datasets** for cross-regime robustness.

This spec covers all four as one cohesive hardening pass. Phase 2 (LLM agents) and its
failure-mode testing (strategic agents, noisy observations, communication constraints)
are explicitly out of scope here, but the communication-graph work is the substrate the
Phase 2 communication-constraint experiments build on.

## Non-goals (YAGNI)

- No LLM agents, memory, reflection, or natural-language messaging (Phase 2).
- No fairness-weighted / needs-weighted welfare model (Phase 3; served-load fraction
  stays the Phase 1.x welfare base, advisor-confirmed).
- No new data *source* beyond NREL ResStock (additional ResStock climate regions only).
- No fairness tiebreak inside the LP objective (noted as a future refinement).
- No changes to the core physics (`household.step`, `settle_transfers` math).

## Section 1 — Communication-graph overlays

### Principle

Coordination edges follow real-world relationships. A household can simultaneously sit
in several **overlapping trust circles**: its geographic neighbors, the portfolio of a
single owner who holds multiple properties, a property-management company's managed set,
a community-solar operator's members, an HOA, a demand-response aggregator's enrolled
homes. Geographic adjacency is *one* layer, retained as a reasonable starting point, and
augmented by an **extensible set of named affiliation layers**.

The communication graph is a **coordination-scope** constraint, not a physical one. The
physical bus connects all 30 houses; `comm_graph` / overlay edges only govern *which
peers a coordination strategy is willing to coordinate with*. `settle_transfers` continues
to enforce the physical rules (bus saturation, no-wheeling between differing grid states)
regardless of overlays. Concretely: a same-owner pair where one house is islanded and the
other grid-connected still cannot wheel grid energy across the bus.

### Data model

- **`Household`** gains `affiliations: dict[str, str]` mapping an affiliation *type*
  (e.g. `"owner"`, `"property_manager"`, `"community_solar"`, `"hoa"`, `"dr_aggregator"`)
  to a *group id* (e.g. `"owner_acme"`). The set of types is open — scenarios define
  whichever types they use. A house omitting a type simply has no edge of that type.
- **`Neighborhood`** (`sim/network.py`) gains
  `edges_by_type: dict[str, dict[str, list[str]]]`, keyed by affiliation type plus the
  reserved key `"geographic"`. Each inner dict is house_id → neighbor list. The existing
  `comm_graph` field is retained and equals the `"geographic"` layer, so existing code
  and the geographic `round_robin` strategy are unchanged.
- **`Neighborhood.union_neighbors(hid) -> list[str]`**: deterministic sorted union of a
  house's neighbors across the geographic layer and every affiliation layer, excluding
  `hid` itself.

### Edge derivation

Within any one affiliation group, all member houses are mutually connected (a clique).
Geographic edges are built exactly as today (`build_grid_neighborhood`, 4-neighbor N/E/S/W).

A new builder `build_overlay_neighborhood(rows, cols, affiliations, *, bus_max_kw, bus_loss_factor)`:
1. builds the geographic layer via the existing logic,
2. for each affiliation type, derives a clique per group,
3. returns a `Neighborhood` with both `comm_graph` (= geographic) and `edges_by_type` populated.

### Scenario YAML surface

Scenarios carry an explicit `affiliations:` block — **explicit, not randomly generated** —
so the trust-circle structure is fully reproducible and deliberately designed:

```yaml
affiliations:
  owner:
    owner_acme: [r0c0, r0c1, r4c5]      # one owner, three scattered properties
    owner_singh: [r2c3, r2c4]
  hoa:
    hoa_north: [r0c0, r0c1, r0c2, r1c0, r1c1, r1c2]
  dr_aggregator:
    agg_gridflex: [r0c0, r3c3, r4c4, r2c1]
```

A helper `default_affiliations(rows, cols, seed)` emits one plausible structure for a
30-house grid (a couple of multi-property owners, one HOA covering a contiguous block,
one demand-response aggregator spanning the grid) for scenarios that don't hand-author
one. Seeded for determinism.

### Loader + validation

`load_scenario` parses the `affiliations:` block into the `Household.affiliations` data
and validates that every referenced house id exists in the `rows × cols` grid.

## Section 2 — `round_robin_overlay` strategy

New module `sim/strategies/round_robin_overlay.py`. Identical share-the-headroom logic to
the existing `round_robin` (share `SHARE_FRACTION` of above-DoD-floor headroom from
above-mean-SoC islanded houses to below-mean targets), but **targets are
`neighborhood.union_neighbors(hid)`** (geographic ∪ all affiliation layers) rather than
geographic-only.

This is the intermediate baseline. It demonstrates — without any LLM — that the overlay
structure carries coordination value: "even naive sharing across ownership/management
edges beats geographic-only sharing." It also gives Phase 2 a meaningful non-trivial
target to beat, between geographic round-robin and the LP ceiling.

The geographic `round_robin` is retained unchanged as the weak baseline so the
geographic-vs-overlay comparison is available.

## Section 3 — Stress-scenario matrix

Four scenarios designed to drive the no-coordination baseline well below its current 98.5%
ceiling and leave round-robin visibly failing.

| Scenario file | Data | Stress mechanism |
|---|---|---|
| `winter_morning_lowsolar.yaml` | cold-climate ResStock region + winter NSRDB | January, low solar yield, 24 h outage over the morning demand peak |
| `heatwave_ac.yaml` | hot-climate ResStock region + summer NSRDB | summer afternoon AC-driven demand peak under outage |
| `long_outage_72h.yaml` | Texas ResStock | 72 h continuous outage (batteries cannot recover) |
| `haves_havenots.yaml` | Texas ResStock | bimodal capacity (see below) under outage |

All four use the overlay affiliation structure from Section 1 and are run against all
strategies (no-coordination, geographic round_robin, round_robin_overlay, LP-optimal).

### Bimodal capacity sampling

`household_sampling` gains `mode: "uniform" | "bimodal"` (default `"uniform"`, preserving
existing scenarios). In `"bimodal"` mode, sampling draws each house from one of two seeded
clusters — a "have" cluster (large PV + battery) and a "have-not" cluster (small or no
PV/battery) — with a configurable mix fraction. Deterministic given `scenario.seed`.

### Repeated-outage variant (free)

`OutageWindow` already supports multiple windows, so "repeated outages with no recovery
window" is expressible by listing several outage windows in any scenario. Not a separate
file unless a stress scenario needs it.

### Acceptance criterion (the advisor's real bar)

At least one scenario must establish a regime where simple sharing visibly breaks. The
concrete, testable bar:

- no-coordination served-load fraction **< 0.90** (well below the current 0.985), **and**
- geographic `round_robin` leaves **unmet kWh that `round_robin_overlay` and/or LP
  measurably recover** (a non-trivial gap to close).

If none of the four scenarios meets this bar, the stress knobs (outage length, solar
derate, demand intensity, capacity split) are tuned until one does, before Phase 1.6 is
declared complete. An automated integration test asserts the bar on the designated
scenario(s).

## Section 4 — Centralized LP optimal baseline (the ceiling)

New module `sim/strategies/lp_optimal.py`. A **full-horizon, full-bus, perfect-foresight
linear program** defining the theoretical upper bound on served load.

### Formulation

- **Decision variables**, per timestep × house:
  - battery charge power and discharge power (separate non-negative variables so the
    round-trip √η efficiency stays linear),
  - inter-house transfer flow on the shared bus,
  - grid import / export (permitted only when the house is grid-connected at that tick),
  - served load and unmet load.
- **Constraints:**
  - battery state-of-charge recurrence across ticks, with √η on each leg,
  - SoC bounds: DoD floor ≤ SoC ≤ capacity,
  - battery charge/discharge rate limits,
  - bus throughput ≤ `bus_max_kw`,
  - no-wheeling during outage: islanded houses receive no grid energy and cannot pass
    grid energy across the bus,
  - served + unmet = demand; all powers non-negative.
- **Objective:** maximize total **served load energy** over the horizon. The LP ignores
  the communication graph entirely (full-bus, full-information), which is what makes it a
  true physical ceiling rather than a topology-bounded one. The LP's resulting welfare
  Gini is reported as-is; a lexicographic fairness second stage (maximize served, then
  minimize welfare inequality among equal-served solutions) is a documented future
  refinement, deliberately deferred.

### Solver

`scipy.optimize.linprog(method="highs")` — deterministic and mature. **This adds `scipy`
as a new dependency**, which trips the project's clean-install CI rule. The implementation
plan must include a clean-install dry-run in `/tmp/microgrid_ci_check` (and a CI green-check
after push) for any commit touching `pyproject.toml`.

### Interface: the `prepare` hook

The existing strategy interface `decide_transfers(t, states, households, solar, load, grid,
neighborhood, dt_hours)` is **myopic** — it cannot see future solar/load, so a foresighted
LP cannot fit it directly.

Add an **optional** `prepare(scenario, households, solar_profile, load_profiles, neighborhood)`
hook. Before the per-tick loop, `engine.run` calls `strategy.prepare(...)` if the strategy
module defines it. The LP strategy uses `prepare` to solve the entire LP once over the full
horizon and cache a per-tick transfer schedule; its `decide_transfers` then returns the
slice for tick `t`. Myopic strategies (`no_coordination`, `round_robin`,
`round_robin_overlay`) do not define `prepare` and are unaffected. This hook is also
forward-useful for Phase 2 LLM agents, which will want a setup phase.

Determinism requirement (project-wide) holds: two runs of the same LP scenario must
produce byte-identical state logs. HiGHS is deterministic; the schedule is cached
deterministically.

## Section 5 — "Gap closed" reporting

The advisor's target framing is *"coordination closes X% of the gap between round-robin and
LP-optimal."* This is a cross-run metric:

```
gap_closed(strategy) = (served(strategy) - served(round_robin))
                       / (served(lp_optimal) - served(round_robin))
```

`summary.json` already carries `served_load_fraction`, `gini_welfare`, `unmet_kwh_total`,
etc. per run (`sim/logging.py:finalize`), so no change to per-run output is needed. A thin
`scripts/compare.py` reads several runs' `summary.json` for one scenario across strategies
and tabulates served-load fraction, unmet kWh, welfare Gini, and gap-closed per strategy as
a markdown table. (This may later be folded into the Phase 3 `/sweep` skill; for 1.6 a
standalone comparison script is sufficient.)

## Testing strategy

TDD throughout (red → minimal green → commit), per project conventions. Notable tests:

- **Overlay graph:** affiliation cliques derived correctly; `union_neighbors` is the sorted
  dedup union; geographic layer unchanged; YAML round-trips and validates bad house ids.
- **round_robin_overlay:** on a hand-built neighborhood with an owner edge between two
  otherwise non-adjacent houses, a transfer occurs across that owner edge that geographic
  round_robin would not make.
- **bimodal sampling:** deterministic given seed; produces two distinct capacity clusters.
- **LP correctness:** on a tiny hand-solvable instance (2 houses, few ticks), the LP's
  served load matches the hand-computed optimum; LP ≥ round_robin_overlay ≥ round_robin ≥
  no_coordination on served load for the same scenario (the ordering the "gap" framing
  depends on).
- **LP determinism:** two runs → byte-identical state logs.
- **prepare hook:** engine calls `prepare` exactly once when defined; myopic strategies
  run unchanged when it is absent.
- **Stress acceptance:** the designated breaking scenario meets the Section 3 bar
  (no-coord < 0.90 served; non-trivial unmet for round_robin that overlay/LP recover).
- **Packaging:** clean-install dry-run in `/tmp/microgrid_ci_check` after adding `scipy`.
